# 提示词管理功能 · 设计讨论纪要

> 本文档记录 v1.0 之后第一个核心特性"提示词工程化"的设计讨论思路，作为后续开发的对齐基线。
> 状态：**讨论中**，话题 1（工作流版本化 + 触发入口）已对齐，话题 2-4 待续。

---

## 一、问题缘起

### 1.1 用户现实痛点

现在工具产出的图片**无法溯源、无法复现、无法精进**：
- AI 生成图高度依赖于"提示词 + 参考图"，但每张产出的图丢了它的"配方"
- 偶然出了好图，下次想复现不知道用的什么提示词
- 无法系统化积累"什么样的打法稳定 work"

### 1.2 进一步定位

最初讨论方向是给每张图打"配方溯源"。但深入聊后定位调整到**更靠根**的问题：

> 提示词目前是散落在工作流里的纯文本字符串，没有身份、没有版本、不能复用、无法对比。**先把提示词从"字符串"提升为"有身份的资产"**，溯源问题就自然解决了——每张图能锚定到"哪条提示词的哪个版本"。

### 1.3 关键洞察（用户提出）

**提示词不能孤立评价好坏，要看"组合行为"**：
- 工作流是线性的，上游 prompt 的产出 → 下游 prompt 的输入
- 上下游各自是好词，组合后可能产生坏结果（契约错位）
- 所以"组合"本身才是真正要管理的资产

## 二、异业参考与启示

| 行业 | 机制 | 启示 |
|---|---|---|
| **软件工程** | semver + lock file | 组合本身要被命名、版本化、可锁定。不是各组件最新就 OK |
| **菜谱体系** | 调料 / 酱汁配方 / 菜谱 三层 | 中间层（酱汁配方）才是大厨真正的经验资产，不是单一调料 |
| **Figma 设计系统** | token / component / pattern | 层次化引用 + 局部锁定，是同时兼顾"复用"和"稳定"的标准解法 |
| **MLflow experiment** | 把 hyperparam 组合记录下来追踪效果 | 经验固化在"组合"层，不在原子层 |

## 三、核心结构：三层模型

```
Atom（原子提示词）           例："用 XX 设计原理分析产品图"
   ↑ 被引用 + 锁定版本
   │
Workflow（流程骨架）         节点 + 边 + 含 N 个 PromptNode 占位
   ↓ 1:1 绑定（Recipe atom 数量必须等于 PromptNode 数量）
Recipe（atom 组合资产）      每个 PromptNode 装哪条 atom 的哪个版本
   ├─ PromptNode 1 → atom1 @ v1
   ├─ PromptNode 2 → atom2 @ v4
   └─ PromptNode N → atom3 @ v2
```

### 3.1 各层定义

**Atom（原子提示词）**
- 单条 prompt（如"用 XX 设计原理分析产品图"）
- 全局资产、独立版本化
- 可被多个 Recipe / 多条工作流引用，每处独立锁版本

**Workflow（流程骨架）**
- 节点 + 边的纯结构
- 仅有 **草稿 / 已发布** 两态，**不做 v1/v2 时间线版本化**
- 想做版本演进 → 复制一条新工作流（新的 ID + 新的 Recipe 命名空间），手动管理
- 工作流通过插入 **PromptNode** 来声明"我有几个 prompt 槽"

**Recipe（atom 组合资产）**
- **绑定到一个具体工作流**，不漂浮在工作流之上、不能跨工作流复用
- 只管"每个 PromptNode 装哪条 atom 的哪个版本"，**不管流程结构（节点连线 / 增删节点等）**
- 一个工作流可以有多个 Recipe（v1 / v2 / 汽车版 / 家电版），用于"同一流程下试不同 atom 组合"
- 独立版本化（v1, v2, ...）
- atom 数量必须 = 工作流当前 PromptNode 数量

### 3.2 装载关系（运行时）

工作流执行时装载一个 Recipe，按 PromptNode 稳定 ID 1:1 注入：

```
Recipe.bindings = {
  [promptNodeId_A]: { atomId, atomVersionId, lockedToVersion: true },
  [promptNodeId_B]: { atomId, atomVersionId, lockedToVersion: false },  // 跟随最新
  ...
}
```

- **`lockedToVersion: true`** → 跑工作流时取 `atomVersionId` 这一固定版本
- **`lockedToVersion: false`** → 跑工作流时取该 atom 的当前最新版本
- 装载是**运行时关系**，不绑死——同一条工作流可装 Recipe A 跑一次、再装 Recipe B 跑一次做 A/B 对比

## 四、节点体系：严格二分

工作流里**所有提示词输入都走两种节点之一**——没有"内部 prompt 字段"：

| 节点类型 | 用途 | 是否资产化 |
|---|---|---|
| **PromptNode**（提示词节点） | 需要资产化、迭代积累的提示词 | ✅ 通过 atom 引用，版本化、跨工作流复用 |
| **TextInputNode**（文本输入节点） | 一次性、工作流特定的边角需求 | ❌ 纯文本，不版本化 |

### 4.1 现有节点的改造

**必须废除的节点内部字段**（所有 prompt 都改走外部连接）：
- `LLMNode.systemPrompt` → 节点改为暴露"system prompt"输入端口，必须由 PromptNode 或 TextInputNode 连入
- `ImageGenNode.negativePrompt` → 节点改为暴露"negative prompt"输入端口
- `DAGEngine.AR_PROMPT_MAP`（比例 → 描述文字的硬编码 prompt）→ 提取为系统级 atoms 预置在 atom 库

**保持原样**：
- `ImageInputNode` / `ImageSaveNode` / `TextDisplayNode` / `SendToCanvasNode`：不涉及 prompt
- `CodeNode.requirement`：是开发期辅助（生成代码用），非运行时 prompt

### 4.2 PromptNode 的形态

**节点上的功能**：
- **选择框**：选已有 atom（搭建工作流时提效）
- **文本框**：载入选中 atom 的内容，支持就地增删改
- **保存动作**：改了内容后必须显式保存——二选一：(a) 作为该 atom 的新版本，(b) 新建一条 atom
- **输出端口**：连到任何接收 prompt 文本的节点

**PromptNode 自身的语义**：
- 它是"工作流里的一个 prompt 槽位"——声明"这里需要一条 atom"
- 是否存"默认 atom"作为裸跑兜底（没装 Recipe 时也能跑）：**待细化，倾向"有默认"**
- 工作流增删 PromptNode 直接影响该工作流下所有 Recipe 的兼容性

### 4.3 现有数据迁移策略

工具升级时：
- 旧 LLMNode 的 systemPrompt 内容 → 自动生成 TextInputNode 接入对应端口（保留原文，不丢）
- 旧 ImageGenNode 的 negativePrompt 内容 → 同上
- 不强制转 atom——用户在管理页主动"提取为 atom"才升级到资产化

## 五、关键设计决策（已对齐）

### 5.1 工作流的版本化

**结论：不做版本化时间线，仅有"草稿 / 已发布"两态。要做 v1/v2 演进 → 复制工作流。**

- 编辑工作流时是草稿
- 主动"发布" → 工作流变为可被画板触发的状态
- 想做 v2 → 复制整条工作流（新 ID），独立演化
- 复制时 PromptNode 和当前装载的 Recipe 配置一起跟随过来

理由：避免"schema diff 迁移向导"的复杂工程，复制工作流足够简单且符合心智模型。

### 5.2 Recipe 的版本化

**结论：主动发布产生新版本。**

- Recipe 改 atom 绑定不自动升版本（避免变成日志而非经验沉淀）
- 用户主动"发布 Recipe v2"——意味着"我验证过这套组合是好的"

### 5.3 PromptNode 增删时 Recipe 怎么处理

**结论：保留 Recipe + 标记"不完整"，让用户主动补齐，不自动作废。**

- 加 PromptNode → 现有 Recipe 缺一个 atom，状态变"不完整"，必须补齐才能跑
- 删 PromptNode → 现有 Recipe 多余的 atom 自动丢弃，无影响
- 改 PromptNode 内容（在节点上改文本→保存）→ 影响当前装载 Recipe 的对应绑定

### 5.4 编辑权限分工

**心智模型：工作流页是"实验沙盒"，提示词管理页是"经验沉淀仓库"。**

| 动作 | 工作流页 | 管理页 |
|---|---|---|
| 选哪个 atom、锁哪个版本 | ✅ | ✅ |
| 编辑 atom 内容 / 发布新版本 | ✅（在 PromptNode 上改+保存为新版本/新 atom） | ✅ |
| 临时调整 Recipe 槽位（试跑用，不持久化） | ✅ | — |
| 永久修改 Recipe（commit） | ❌ | ✅ |
| "另存为新 Recipe" | ✅ | ✅ |

### 5.5 上下游契约描述（Atom 元数据里写"我期望什么输入 / 输出什么格式"）要做吗？

**结论：v1 不做。**

- Recipe 这一层已沉淀"经验验证过的组合"
- atom 升级时"跑一遍新 Recipe 看效果"就能兜底
- 等 atom > 30、Recipe > 5 后再加自然语言版契约描述

### 5.6 变量化（模板 + 变量绑定）

**结论：v1 不做，先做纯文本 atom + 版本。**

变量化等真出现"多条工作流共用同一套 prompt 模板但参数不同"的明确需求再做。

### 5.7 版本锁默认值

**结论：默认锁定到"当前最新版本"，"跟随最新"作为显式选择。**

避免设计师没意识到 atom 升级会影响这条工作流的稳定性。

### 5.8 溯源机制

**结论：每次工作流执行存一份 snapshot。**

工作流不版本化的代价是"今天的工作流改了之后昨天跑过的好图怎么原样复现"。解法：

- 每次执行（task 记录）增加一个 `snapshot` 字段
- 内容：本次跑的工作流 nodes + edges + 各 PromptNode 解析后的 atom 内容（直接存文本，不存引用）
- snapshot 是 immutable 的，不依赖工作流当前状态

## 六、话题 1 已对齐：工作流版本化 + 画板触发入口

### 6.1 工作流版本化

详见 5.1——草稿/发布两态，复制工作流代替版本演进。

### 6.2 画板"发送到工作流"触发入口的演变

当前：选图后弹框，仅选"哪个工作流"。

新版：

```
┌─ 发送到工作流 ────────────┐
│ 工作流：CMF 替换  ▼       │
│ 配方：  Recipe v2 ▼       │   ← 默认是该工作流的"默认 Recipe"
│                          │
│ ▶ 高级：手动调整槽位绑定  │   ← v1 不做，置灰或隐藏
│                          │
│ [取消]    [发送]          │
└──────────────────────────┘
```

关键 UX 决策：
- **每个工作流有"默认 Recipe"概念**——90% 场景一键发送，不用选
- 选了不同 Recipe 后入口"记忆"——下次默认用这个
- A/B 对比通过"再发送一次，换 Recipe"实现，不在弹框里做并行

## 七、这套机制如何回应原始痛点

| 用户诉求 | 这套机制怎么回应 |
|---|---|
| 同一条 atom 在不同工作流用不同版本 | 不同工作流的 PromptNode 通过 Recipe 锁不同版本 |
| 上下游 prompt 单看好但组合差 | Recipe 是"经验组合"的命名快照，发布新 Recipe 而非改单条 atom |
| 想做效果对比 | 同一工作流装不同 Recipe 平行跑 |
| 想溯源好图怎么来的 | 每次执行存 snapshot（含所有 PromptNode 的 atom 内容），永久可复现 |
| 经验积累 | atom 库（基础调料） + Recipe 库（大厨的酱汁配方）双层沉淀 |
