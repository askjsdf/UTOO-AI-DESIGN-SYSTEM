### 3.3 全库扫描函数（跨文件夹查询）

"全部"、"最近添加"、"未分类"、颜色过滤等功能需要扫描所有文件夹。

```typescript
// 递归扫描 OPFS 库根目录下所有文件
export async function scanAllFiles(
  root: FileSystemDirectoryHandle,
  maxDepth = 6
): Promise<LibraryFile[]>
```

**性能策略**：
- 首次加载异步扫描，显示进度
- 结果缓存在 `useLibraryStore` 内存中
- 文件操作后（上传/删除/重命名）增量更新缓存，不重新全量扫描
- 单次扫描通常 < 200ms（OPFS 是内存级速度）

## 四、服务层设计

### 4.1 颜色提取（新增 `extractDominantColors`）

位置：`LibraryFileService.ts` 或新文件 `src/utils/colorExtract.ts`

```typescript
/**
 * 从图片 Blob 提取 5 个主色调（使用 canvas 降采样 + 颜色量化）
 * 返回 HEX 字符串数组，长度 0-5
 */
export async function extractDominantColors(blob: Blob): Promise<string[]>
```

**算法**（无需第三方库）：
1. 创建 `OffscreenCanvas(64, 64)`（Worker 内可用）或 `<canvas>` DOM
2. 将图片绘制到 64×64（下采样减少计算量）
3. 读取 `getImageData` 所有像素
4. 按 HSL 色相区间分成 18 个桶（每 20 度一档）
5. 取各桶像素平均值，选出权重最高的 5 个桶
6. 转换为 HEX 返回

**调用时机**：
- `writeFile()` 写入后，若是图片类型，异步调用提取并 patch `fileColors`
- 已存在文件首次显示时，若 `colors` 为空，后台触发提取

### 4.2 标签聚合（新增 `getAllTags`）

## 五、状态管理层设计

### 5.1 独立 Library Store

不进入 `appStore.ts`，单独创建 `src/store/libraryStore.ts`，用 Zustand 管理。

```typescript
// src/store/libraryStore.ts

/** 导航目标类型 */
export type NavTarget =
  | { type: 'folder'; handle: FileSystemDirectoryHandle; path: string[] }
  | { type: 'all' }
  | { type: 'recent' }     // 最近 30 天
  | { type: 'untagged' }
  | { type: 'starred' }    // rating > 0
  | { type: 'color'; hex: string }   // 颜色过滤
  | { type: 'tag'; tag: string }     // 标签视图点击

/** 排序配置 */
export interface SortConfig {
  field: 'name' | 'addedAt' | 'size' | 'rating'
  dir:   'asc' | 'desc'
}

/** 过滤配置 */
export interface FilterConfig {
  formats:   string[]    // ['jpg', 'png'] 空数组=不过滤
  minRating: number      // 0=不过滤, 1-5=最低星级
  tags:      string[]    // 多标签 AND
}

/** 视图配置 */
export interface ViewConfig {
  mode:      'grid' | 'list'
  thumbSize: number      // 100-400, 默认 200
}

interface LibraryState {
  // ── 导航 ──────────────────────────────────
  navTarget: NavTarget | null
  setNavTarget: (t: NavTarget | null) => void

  // ── 当前视图文件列表（已排序+过滤） ────────
  files: LibraryFile[]
  isLoading: boolean
  loadFiles: () => Promise<void>   // 根据 navTarget 重新加载

  // ── 选中文件 ──────────────────────────────
  selectedFile: LibraryFile | null
  setSelectedFile: (f: LibraryFile | null) => void

  // ── 排序/过滤/视图配置 ─────────────────────
  sort: SortConfig
  setSort: (s: SortConfig) => void
  filter: FilterConfig
  setFilter: (f: FilterConfig) => void
  view: ViewConfig
  setView: (v: Partial<ViewConfig>) => void

  // ── 标签聚合（全库） ───────────────────────
  allTags: Map<string, number>
  refreshAllTags: () => Promise<void>

  // ── 文件操作（含状态更新） ─────────────────
  uploadFiles: (folder: { handle: FileSystemDirectoryHandle; path: string[] }, files: File[]) => Promise<void>
  deleteFile: (file: LibraryFile) => Promise<void>
  renameFile: (file: LibraryFile, newName: string) => Promise<void>
  updateTags: (file: LibraryFile, tags: string[]) => Promise<void>
  updateRating: (file: LibraryFile, rating: number) => Promise<void>
  updateNote: (file: LibraryFile, note: string) => Promise<void>
}
```

**关键设计点**：
- `loadFiles()` 根据 `navTarget` 决定是调 `listFiles(folder.handle)` 还是调 `scanAllFiles(root)`
- 文件操作（增删改）在完成后**增量更新** `files` 数组，不重新 `loadFiles()`，保证操作流畅
- `sort` 和 `filter` 变化时直接对内存中的 `files` 重新排序/过滤，不重新读 OPFS
- `allTags` 在 Library 模块挂载时异步加载一次，标签变化时 invalidate

### 6.3 JustifiedGrid（等高行布局）

Eagle 默认的 Justified Grid 是：**每行图片等高，宽度按原始比例自适应填满行宽**。

实现方案：**CSS Flex + 宽高比计算**，不依赖第三方库。

```typescript
// JustifiedGrid.tsx
interface JustifiedGridProps {
  files: LibraryFile[]
  thumbSize: number       // 行高基准值（100-400px），对应缩略图大小滑条
  gap: number             // 间距
  selectedFile: LibraryFile | null
  onSelectFile: (f: LibraryFile) => void
}

// 计算逻辑：
// 1. 若 dimensions 已知：aspectRatio = w/h
// 2. 若 dimensions 未知（图片未加载）：默认比例 = 1（正方形占位）
// 3. CSS: 每个 <div> 设置 flex-grow = aspectRatio
//         height = thumbSize (px) 固定行高
//         width = 自适应

// 核心 CSS 模式：
// .row { display: flex; gap: 4px; height: {thumbSize}px; }
// .card { flex-grow: {ratio}; min-width: 50px; overflow: hidden; }
// .card img { width: 100%; height: 100%; object-fit: cover; }
```

**首次加载时 dimensions 可能为空的降级策略**：
- 图片在 `GridCard` 的 `onLoad` 回调中读取 `naturalWidth / naturalHeight`
- 通知 store 更新 `dimensions`，同时调 `setFileDimensions()` 写入 `_meta.json`
- 下次加载直接读取缓存，立即使用正确比例，不再抖动

**缩略图大小滑条**：

`thumbSize` 对应 JustifiedGrid 的行高。滑条范围 `[80, 400]`，默认 `200`。
- 存入 `useLibraryStore.view.thumbSize`（内存状态，不持久化）
- 改变时直接重渲染，无 OPFS 操作

### 6.7 NoteEditor 持久化版本

// NoteEditor.tsx
// Textarea + debounce 500ms 写入 OPFS
function NoteEditor({ file }: { file: LibraryFile }) {
  const updateNote = useLibraryStore((s) => s.updateNote)
  const [text, setText] = useState(file.note)

  const debouncedSave = useMemo(
    () => debounce((t: string) => updateNote(file, t), 500),
    [file.name]
  )

  return (
    <textarea
      value={text}
      onChange={(e) => { setText(e.target.value); debouncedSave(e.target.value) }}
      placeholder="添加备注…"
    />
  )
}

## 八、关键数据流详解

### 8.1 文件上传流

```
用户拖入文件
  → libraryStore.uploadFiles(folder, files)
    → for each file:
        1. 检查重名，生成唯一文件名
        2. LibraryFileService.writeFile(dir, name, buffer, { triggerColorExtract: true })
           ↓ 内部：
           a. OPFS 写入文件
           b. 更新 _meta.json: fileAddedAt[name] = Date.now()
           c. 若是图片：extractDominantColors(blob) → setFileColors(dir, name, colors)
        3. 构造 LibraryFile 对象追加到 store.files（乐观更新）
    → store.allTags invalidate → refreshAllTags()
```

### 8.2 导航切换流

```
用户点击"全部"
  → setNavTarget({ type: 'all' })
  → loadFiles()
    → navTarget.type === 'all':
        scanAllFiles(libraryRoot)  // 递归扫描所有文件夹
        → 填入 store.files（未排序/过滤的原始列表）
  → UI 自动重渲染（Zustand 响应式）
  → sortedFiles = sort(files, sort)
  → filteredFiles = filter(sortedFiles, filter)
  → JustifiedGrid 渲染 filteredFiles
```

### 8.3 颜色搜索流

```
用户在 Inspector 点击色块 #2B5FB3
  → setNavTarget({ type: 'color', hex: '#2B5FB3' })
  → loadFiles()
    → navTarget.type === 'color':
        scanAllFiles(libraryRoot)
        → 过滤: file.colors.some(c => isColorSimilar(c, hex, 35))
        → 按相似度排序（最相似的在前）
  → 左侧导航树"颜色筛选"高亮
```

### 8.4 dimensions 懒加载流

```
JustifiedGrid 渲染 LibraryFile[]
  → 对于 dimensions === null 的图片：
      使用默认比例 1.5（16:9 接近平均值），用于计算初始布局
  → GridCard 挂载后 img.onLoad 触发：
      dims = [naturalWidth, naturalHeight]
      → store.patchFileDimensions(file, dims)   // 更新内存
      → setFileDimensions(dir, file.name, dims) // 写 _meta.json
  → JustifiedGrid 收到 files 变化 → 重新计算布局（平滑过渡）
```

## 十、技术风险与应对

| 风险 | 概率 | 影响 | 应对方案 |
|------|------|------|---------|
| `OffscreenCanvas` 在某些浏览器不可用 | 低 | 颜色提取失败 | fallback 到普通 `<canvas>` DOM API |
| 大量图片首次扫描时 scanAllFiles 慢 | 中 | "全部"视图加载慢 | 显示加载进度条；后台扫描时先显示已加载的批次 |
| dimensions 懒加载导致 Justified Grid 布局抖动 | 中 | 视觉抖动 | 使用 CSS `transition: width 0.2s ease` 平滑过渡 |
| _meta.json 频繁写入（备注 debounce）性能 | 低 | OPFS 写入累积 | debounce 500ms 已足够；OPFS 写入比 IndexedDB 快 |
| 标签自动补全搜索慢（标签量大） | 低 | 输入响应慢 | `allTags` 是 `Map`，查找 O(n)；n<10000 问题不大 |

## 十一、开发顺序建议

```
Phase A（数据基础）
  → 不触动 UI，风险最低，先打好地基
  → 完成后：所有字段可读写，但 UI 还是旧的

Phase B（状态管理）
  → 创建 libraryStore，但还用旧 UI 消费它
  → 完成后：状态层和数据层对齐

Phase C1+C2（网格视图重构）
  → 视觉变化最大，用户最能感知
  → 建议先做，快速获得视觉反馈

Phase C3（固定导航激活）
  → 依赖 Phase A 的 scanAllFiles

Phase C4+C5（颜色提取+搜索）
  → 相对独立，可并行开发

Phase D（P1 功能）
  → 评分/备注/标签补全/排序/过滤
  → 相互独立，可按优先级逐个推进
```
