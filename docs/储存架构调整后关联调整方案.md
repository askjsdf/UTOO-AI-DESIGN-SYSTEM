# 储存架构调整后关联调整方案

> 版本：v1.0  日期：2026-04-10  
> 背景：将画布图片从 base64-in-IndexedDB 迁移到 OPFS 后，系统多处业务流产生了显性和隐性问题。  
> 本文档覆盖：问题台账、架构设计、全生命周期管理、实施方案。

---

## 一、储存层定义

### 1.1 当前系统的五个储存层

```
┌──────────────────────────────────────────────────────────────────┐
│  层级             用途                         键值示例            │
├──────────────────────────────────────────────────────────────────┤
│  localStorage    设置、API Key、迁移标记         utoo_settings     │
│  utoo-engine IDB 工作流、任务、画布项目、对话     tasks / workflows │
│  utoo-canvas IDB tldraw 画布（每项目一个DB）     TLDRAW_DOCUMENT_v2utoo-canvas-{id} │
│  OPFS            图片二进制文件                  canvas-images/{sanitized_assetId}.{ext} │
│  Zustand（内存）  运行时状态、待写入队列           pendingCanvasImages │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 各层职责边界（目标状态）

| 层级 | 存什么 | 不存什么 |
|------|--------|---------|
| localStorage | 轻量配置 | 任何图片数据 |
| utoo-engine IDB | 结构数据（任务元信息、工作流拓扑、项目列表） | 全尺寸 base64 图片 |
| utoo-canvas IDB | tldraw 形状/页面/asset 记录（src 为 URL） | base64 图片内容 |
| OPFS | 所有持久化图片二进制文件 | 文本/配置 |
| Zustand | 执行期间的临时图片（base64 ephemeral） | 跨会话数据 |

## 二、图片数据格式规范

### 2.1 格式使用规则

```
base64（ephemeral）
  ├─ 工作流引擎内部节点间传递  ✅
  ├─ CanvasSidebar 附图（session-local）  ✅
  ├─ ImageInputNode 持有的输入图（仅执行期）  ✅
  ├─ 任务 inputPreviews / outputPreviews  ❌ 改为 OPFS URL 或缩略图
  ├─ 工作流定义 SavedNode.data.images  ❌ 禁止持久化
  └─ tldraw asset.props.src  ❌ 禁止

/__local_asset__/{encodedAssetId}
  ├─ tldraw asset.props.src  ✅（唯一允许的持久化图片引用）
  └─ 任务 outputPreviews（指向 OPFS 文件）  ✅

OPFS 二进制文件（canvas-images/{sanitized_assetId}.{ext}）
  └─ 所有需要持久化的图片内容  ✅
```

### 2.2 缩略图规范（用于 UI 预览）

任务卡片等 UI 预览允许使用 base64，但必须是**缩略图**：

- 最大尺寸：256 × 256 px
- 格式：JPEG，质量 0.7
- 最大体积：< 50KB
- 生成方式：OffscreenCanvas resize 后转 base64

## 三、图片全生命周期管理

### 3.1 生命周期总览

```
[创建] ──→ [持久化到 OPFS] ──→ [tldraw 引用 URL] ──→ [SW 服务]
                                        │
                                   [用户操作]
                                   /    |    \
                              [删除]  [导出]  [项目删除]
                                │
                         [GC 标记 → 延迟清理]
```

### 3.2 入口：图片如何进入系统

**当前状态与目标对比：**

| 入口 | 当前行为 | 目标行为 |
|------|---------|---------|
| AI 生成（DAGEngine imageGen） | base64 在引擎内流转，sendToCanvas 时存 OPFS ✅ | 维持现状 |
| 画布侧边栏 AI 生成 | 直接存 OPFS ✅ | 维持现状 |
| ImageInputNode 文件选择 | base64 在节点内存中（ephemeral）✅ | 维持现状 |
| 文件拖拽到画布 | tldraw 原生处理 → base64 进 IDB ❌ | 拦截后存 OPFS |
| Figma/剪贴板粘贴 | tldraw 原生处理 → base64 进 IDB ❌ | 拦截后存 OPFS |
| tldraw 自带"上传媒体"按钮 | tldraw 原生处理 → base64 进 IDB ❌ | 拦截后存 OPFS |

**拦截方案（registerExternalAssetHandler）：**

在 `CanvasView/index.tsx` 的 `handleMount` 中注册：

```typescript
// handleMount 内部追加
editor.registerExternalAssetHandler('file', async ({ file }) => {
  const assetId = AssetRecordType.createId()
  // File → DataURL → OPFS → local URL
  const dataUrl = await new Promise<string>((resolve, reject) => {
    const reader = new FileReader()
    reader.onload = () => resolve(reader.result as string)
    reader.onerror = reject
    reader.readAsDataURL(file)
  })
  const localUrl = await saveImageDataUrl(assetId, dataUrl)
  const { w, h } = await getImageDimensions(localUrl)
  return {
    id: assetId,
    type: 'image',
    typeName: 'asset',
    props: {
      name: file.name,
      src: localUrl,   // ← /__local_asset__/... 而非 base64
      w, h,
      mimeType: file.type || 'image/png',
      isAnimated: false,
    },
    meta: {},
  }
})
```

此钩子覆盖：
- 从文件管理器拖拽到画布
- 剪贴板粘贴（Ctrl+V，包括 Figma 复制的 PNG）
- tldraw 工具栏"上传媒体"按钮

### 3.3 更新：图片写入画布

**路径 A：AI 生成 → 插入（无占位符）**
```
execSendToCanvas
  → addPendingCanvasImages(projectId, dataUrls)
  → CanvasView.useEffect[pendingCanvasImages]
  → insertImagesIntoEditor（await + try-catch）
  → saveImageDataUrl → OPFS
  → createAssets(src = /__local_asset__/...)
  → createShape
```

**路径 B：AI 生成 → 替换占位符**
```
execSendToCanvas（有 canvasPlaceholderId）
  → addPendingCanvasReplacement(projectId, placeholderId, dataUrls)
  → CanvasView.useEffect[pendingCanvasReplacements]（已 await）
  → replacePlaceholderWithDataUrls（try-catch 包裹）
  → deleteShape(placeholder)    ← 同步删除
  → saveImageDataUrl → OPFS    ← 异步（失败时保留备用图）
  → createAssets + createShape
```

**路径 B 的占位符保护机制（新增）：**
```
任务开始时：appStore 记录 taskId → placeholderShapeId 映射
任务失败时：自动删除该 taskId 对应的所有占位符
任务完成时：替换后清除映射记录
```

### 3.4 删除：图片从画布移除

**场景 1：用户手动删除单张图片 shape**

tldraw 会从 IDB 中删除 shape 和 asset 记录，但 **OPFS 文件不会自动删除**。  
直接监听并立即删除 OPFS 文件是不安全的（用户可能 Ctrl+Z 撤销）。

**方案：GC 垃圾回收机制（延迟清理）**

**场景 3：用户在画布内 Ctrl+Z 撤销（图片恢复）**

由于使用 GC 延迟清理而非即时删除，撤销后 OPFS 文件仍然存在，tldraw 恢复 asset 记录后图片可正常显示。**无需额外处理。**

**场景 4：画布项目复制（如有）**

复制项目时，新项目的 tldraw IDB 会包含相同的 asset 记录（相同 assetId）。OPFS 文件被两个项目共享引用。GC 时由于两个项目都引用该文件，不会误删。

### 3.5 导出：图片用于输出

| 导出类型 | 图片来源 | 依赖 SW | 风险 |
|---------|---------|---------|------|
| tldraw exportToBlob（画布→工作流） | SW 服务 /__local_asset__/ → tldraw 渲染 | 是 | SW 未就绪时导出空图 |
| imageSave 节点保存文件 | base64 直接写文件 | 否 | 无 |
| 画布快照/截图 | 同 exportToBlob | 是 | 同上 |
| OPFS 批量导出（recover.html） | 直接读 OPFS 文件 | 否 | 无 |

### Bug #2 🔴：批量任务 OOM 崩溃（50个任务场景）

**位置**：`DAGEngine.ts`（execImageSave、execImageGen）、`appStore.ts`（_runNextTask）

**修复 A：execImageSave 改存 OPFS URL**

```typescript
async function execImageSave(node, upstream, patchNodeData): Promise<NodeOutput> {
  // ...原有文件保存逻辑不变...

  const previewUrls: string[] = []
  for (let i = 0; i < Math.min(images.length, 4); i++) {
    const previewId = AssetRecordType.createId() + '_preview'
    const url = await saveBase64Image(previewId, images[i])
    previewUrls.push(url)  // /__local_asset__/... 而非 base64
  }
  patchNodeData(node.id, { savedCount: images.length, savedPreviews: previewUrls })
  return {}
}
```

```typescript
// workflowSend.ts / WorkflowView/index.tsx 修改
// 生成缩略图替代全尺寸 base64
async function createThumbnail(b64: string, maxSize = 256): Promise<string> {
  return new Promise((resolve) => {
    const img = new Image()
    img.onload = () => {
      const scale = Math.min(1, maxSize / Math.max(img.width, img.height))
      const canvas = new OffscreenCanvas(
        Math.round(img.width * scale),
        Math.round(img.height * scale)
      )
      const ctx = canvas.getContext('2d')!
      ctx.drawImage(img, 0, 0, canvas.width, canvas.height)
      canvas.convertToBlob({ type: 'image/jpeg', quality: 0.7 }).then(blob => {
        const reader = new FileReader()
        reader.onload = () => resolve(reader.result as string)
        reader.readAsDataURL(blob)
      })
    }
    img.src = `data:image/png;base64,${b64}`
  })
}

// 用法
```

### Bug #6 🟡：SW 激活窗口期导致 /__local_asset__/ 请求 404

**位置**：`main.tsx`、`App.tsx`

**原因**：`navigator.serviceWorker.register()` 是异步的，不等待激活就渲染 App。虽然 App.tsx 在迁移检查时等待 SW，但迁移已完成的情况下等待时机太晚。

**修复**：在 main.tsx 确保 SW 激活后再挂载 React 应用

```typescript
// main.tsx 修改
async function bootstrap() {

        new Promise(resolve => setTimeout(resolve, 3000)),

    }
  }
  
  ReactDOM.createRoot(document.getElementById('root')!).render(
    <React.StrictMode><App /></React.StrictMode>
  )
}
```

## 五、OPFS 垃圾回收（GC）设计

### 5.1 触发时机

| 时机 | 条件 |
|------|------|
| 应用启动时 | 上次 GC 距今 > 24 小时 |
| 删除画布项目后 | 立即针对该项目的 assetId 运行 |
| 用户主动触发 | recover.html 提供按钮 |

### 5.2 GC 算法

```
1. 列出 OPFS canvas-images/ 中所有文件名（sanitized_assetId.ext）
2. 打开所有 utoo-canvas-* IDB，遍历 records store
   - 找 typeName === 'asset' 且 type === 'image' 的记录
   - 提取 props.src 中的 assetId（从 /__local_asset__/{encodedId} 解码）
   - sanitizeId(assetId) → 文件名前缀
3. 找出 OPFS 文件中不在步骤 2 结果中的 → 标记为孤立
4. 删除孤立文件（仅删除创建时间 > 1 小时的，避免删除刚写入尚未 commit 的文件）
```

### 5.3 GC 不负责的场景

- tldraw 内部 undo/redo 期间的临时删除（GC 延迟机制天然保护）
- `_preview` 后缀的 OPFS 文件（任务预览）：由任务清理时负责
- 迁移期间写入的损坏文件：由 recover.html 单独处理

## 七、架构约束备忘

1. **base64 不得写入任何 IDB**（utoo-engine 或 utoo-canvas）
2. **tldraw asset.props.src 只允许两种值**：`/__local_asset__/...` 或 SVG data URL（< 5KB 的占位符）
3. **任务预览图**：inputPreviews < 50KB/张（缩略图），outputPreviews 使用 OPFS URL
4. **工作流定义**：节点结构可保存，图片数据必须清空
5. **OPFS 文件命名**：统一使用 `sanitizeId(assetId).{ext}`，与 SW 保持一致
6. **GC 安全边界**：只删除创建时间超过 1 小时的孤立文件
7. **SW 依赖**：任何需要渲染/导出 OPFS 图片的操作前，必须确认 SW 激活

### A. 从 tldraw IDB 收集项目的所有 assetId

```typescript
async function collectProjectAssetIds(projectId: string): Promise<string[]> {
  const dbName = `TLDRAW_DOCUMENT_v2utoo-canvas-${projectId}`
  const assetIds: string[] = []
  
  try {
    const db = await new Promise<IDBDatabase>((res, rej) => {
      const req = indexedDB.open(dbName)
      req.onsuccess = () => res(req.result)
      req.onerror = () => rej(req.error)
    })
    
    await new Promise<void>((res, rej) => {
      const tx = db.transaction(['records'], 'readonly')
      const store = tx.objectStore('records')
      const req = store.openCursor()
      req.onsuccess = (e) => {
        const cursor = (e.target as IDBRequest<IDBCursorWithValue>).result
        if (!cursor) { res(); return }
        const record = cursor.value
        if (
          record?.typeName === 'asset' &&
          record?.type === 'image' &&
          typeof record?.props?.src === 'string' &&
          record.props.src.startsWith('/__local_asset__/')
        ) {
          const encoded = record.props.src.slice('/__local_asset__/'.length)
          assetIds.push(decodeURIComponent(encoded))
        }
        cursor.continue()
      }
      req.onerror = () => rej(req.error)
    })
    
    db.close()
  } catch { /* 数据库不存在则忽略 */ }
  
  return assetIds
}
```

```typescript
// imageStore.ts 新增
export async function deleteImageIfOrphaned(
  assetId: string,
  excludeProjectId?: string
): Promise<boolean> {
  // 检查除 excludeProjectId 外的其他项目是否还引用此 assetId
  const allDbs = await indexedDB.databases()
  const otherDbs = allDbs.filter(d =>
    d.name?.includes('utoo-canvas') &&
    !d.name?.includes(excludeProjectId ?? '__none__')
  )
  
  for (const dbInfo of otherDbs) {
    // 简化：任何其他项目存在就不删（精确版本走 GC）
    const ids = await collectProjectAssetIds(
      dbInfo.name!.replace(/.*utoo-canvas-/, '')
    )
    if (ids.includes(assetId)) return false
  }
  
  await deleteImage(assetId)
  return true
}
```
