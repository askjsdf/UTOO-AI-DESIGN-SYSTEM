# 香水设计自动化出图引擎 — 现状分析与整合方案

> 文档时间：2026-04-09  
> 分析范围：ai studio源码 内 4 个原型工程  
> 目标：将四条独立工作流融合为一个平台化、工程化的一键出图系统

---

## 一、四个原型现状速览

### 1. Nocturne KV v5.1 — 主视觉（KV）生成引擎

**定位：** 生成广告级 KV 主视觉图，16:9 / 4K

**工作流（3步串行）：**
```
输入：瓶型图（×N）+ 美学参考图（Aesthetic Anchors）
  ↓
Step 1 analyzeSubject()       → gemini-3.1-pro-preview
        分析瓶子的"Visual DNA"（几何本质 / 材质叙事 / 抽象情绪）
        输出：diagnosis 文字

  ↓
Step 2 generatePrompts()      → gemini-3.1-pro-preview
        结合 diagnosis + 参考图，生成 4 个场景策略 JSON
        4 维度：极简 / 材质回声 / 动态抽象 / 建筑融合

  ↓
Step 3 renderKV() ×4          → gemini-3.1-flash-image-preview
        每个场景策略 + 原图 + 参考图 → 生成 1 张 KV 图
        共输出 4 张 16:9 KV
```

**存储：** IndexedDB（idb-keyval），最多 50 条历史，纯本地

**关键参数：**
- 输入：最多 20 张主体图 + 任意参考图
- 输出：每张主体图 → 4 张 KV
- API 调用次数：1（分析）+ 1（策略）+ 4（渲染）= **6次/张**

**当前问题：**
- 依赖 `window.aistudio` API Key 注入，无法在普通浏览器运行
- 历史记录存本地，无跨设备同步
- 无批量进度可视化（逐个处理，UI 反馈粗糙）

---

### 2. Detail Matrix Engine v2.0 — 细节矩阵引擎

**定位：** 生成"画中画"编辑矩阵图，展示产品局部细节

**工作流（2步串行）：**
```
输入：单张瓶型图
  ↓
Step 1 analyzeProductStyle()  → gemini-3.1-pro-preview
        解析：设计风格 / 底座形式与材质 / 光照情绪 / 色板
        输出：StyleAnalysis JSON 结构体

  ↓
Step 2 generateDetailMatrix() → gemini-3.1-flash-image-preview
        构建 PiP（画中画）prompt：
          中心：产品 + 自适应底座
          外围：3-4 个浮动细节特写面板（100mm 微距感）
        输出：1 张 16:9 矩阵图（1K）
```

**存储：** Firebase Firestore（云端）+ Google Auth（多用户隔离）

**特有能力：**
- **pHash 感知去重**：相同图片不重复处理
- **实时状态写回 Firestore**：`currentStep` 字段驱动前端进度
- **多用户支持**：Google 登录，数据按 uid 隔离
- **档案面板**：历史记录跨设备同步

**关键参数：**
- 输出分辨率：1K（代码注释说明为兼容性考虑暂时降为 1K）
- API 调用次数：1（分析）+ 1（生成）= **2次/张**

**当前问题：**
- 严重依赖 Firebase 配置（`firebase-applet-config.json`），不能直接脱离
- 输出分辨率是 1K，明显低于其他引擎的 4K
- 没有参考图输入通道

---

### 3. Intellectual Moodboard Engine v4.0 — 智识情绪板引擎

**定位：** 生成四维正交情绪板，用于前期设计方向探索

**工作流（2步，含二阶精炼）：**
```
输入：主体图（×N）+ 参考情绪图（可选）
  ↓
Step 1 analyzeProduct()       → gemini-3.1-pro-preview
        系统提示：世界级奢侈品视觉总监人设
        解构 4 个正交轴向：
          · 结构拓扑（轮廓/几何/盖子角度）
          · 材质炼金（玻璃折射/液态互动/渐变）
          · 细节工艺（喷嘴/雕刻/触感微距）
          · 方案演进（多环境/结构变体迭代）
        每个轴向输出：axis / prompt / negativePrompt / hexColors

  ↓
Step 2 generateMoodboard() ×4 → 两阶段精炼
        阶段 A：gemini-2.5-flash-image → 基础情绪板（多面板拼贴）
        阶段 B：gemini-3.1-flash-image-preview → 4K 精炼
        （阶段 B 失败时自动降级回阶段 A 的结果）
```

**存储：** IndexedDB（idb-keyval），按 Session 结构存储（含原图 + 资产）

**特有能力：**
- **两阶段精炼**：唯一采用 Flash→Pro 双模型管线的引擎
- **参考图多张输入**：influencing 分析阶段的 prompt 生成
- **Session 管理**：历史按产品分组，可回放
- **瀑布流展示**：动态渲染，含 prompt 信息和色板

**关键参数：**
- 每张主体图 → 4 张情绪板（每张含多面板拼贴布局）
- API 调用次数：1（分析）+ 4×2（双阶段）= **9次/张**（最重）

**当前问题：**
- 每张图 API 调用最多，成本最高
- 无产品描述文字输入通道（全靠图像理解）
- 二阶精炼偶发失败时降级逻辑静默，用户无感知

### 4. Order & Abundance CMF Narrative Generator v1.0 — CMF叙事批量引擎

**定位：** 批量生成同一产品的 5 种 CMF（色材工艺）变体，4 种构图叙事

**工作流（单步，最简）：**
```
输入：产品图（×N）+ 文字描述（可选）
  ↓
generateCMFNarrative()        → gemini-3.1-flash-image-preview
  每张图 × 4 种布局叙事 = 4 次调用
  每次调用在单张图中合成 5 个 CMF 变体：
    · 原始 CMF（上传图原色）
    · 暖色高光（拉丝金）
    · 冷色高科技（磨砂钛金）
    · 高调不透明（哑光极地白）
    · 低调浓郁（高光黑曜石）

  4 种布局叙事：
    P1 Linear Order   线性排列（工业精度）
    P2 Cluster&Depth  聚集景深（家族感）
    P3 Abstract Data  解构微距（材质信息）
    P4 Radial Spectrum 放射色谱（动态感）

  容错：3.1-flash 报错自动降级 gemini-2.5-flash-image
```

**存储：** localStorage，最多 10 条（受浏览器配额限制有降级逻辑）

**关键参数：**
- 输出：1 张图 → 4 张（每张含 5 个 CMF 变体）
- API 调用次数：**4次/张**（最轻量）

**当前问题：**
- localStorage 存 base64 图片极易超配额（已有三级降级逻辑但根本不够用）
- 没有参考图通道，CMF 方向完全由 prompt 固定，不可自定义
- 无分析阶段，没有针对具体产品的 CMF 个性化

## 三、现有代码的共性结构

四个原型的代码结构惊人一致，抽象后是同一套骨架：

```
App.tsx
  ├── 状态管理（useState）
  │     ├── 输入：sourceImages / subjects / productFiles
  │     ├── 参考：anchors / referenceFiles
  │     ├── 输出：assets / results / items
  │     └── 流程：status / currentStep / progress
  │
  ├── 文件处理（FileReader → base64）
  │
  ├── 主处理函数（for 循环串行）
  │     └── for each 输入图：
  │           Step A: analyzeXxx(image) → 结构化分析
  │           Step B: generateXxx(analysis) ×N → 图像生成
  │
  ├── 存储（IndexedDB / localStorage / Firebase）
  │
  └── UI展示（上传区 / 进度 / 结果网格 / 历史面板）

lib/gemini.ts
  ├── fileToGenerativePart() 或 base64 处理
  ├── analyzeXxx()  → Pro 文本模型
  └── generateXxx() → Flash Image 模型
```

**关键观察：**
- 所有引擎的 `analyzeXxx()` 函数接口完全可以统一：输入 base64 图像，输出结构化 JSON
- 所有引擎的 `generateXxx()` 函数接口完全可以统一：输入分析结果，输出 base64 图像
- 四个引擎的状态机（idle → analyzing → generating → completed → error）是同一模型
- 存储方案四个各不同，是最大的技术债

## 四、整合的核心挑战

### 4.1 数据模型不统一

四个原型各自定义了 `asset` 类型，字段名和结构都不同：

```typescript
// KV引擎
PerfumeAsset { id, file, preview, status, diagnosis, results[], prompts[] }

// 细节矩阵
PortfolioItem { id, uid, originalImage, pHash, detailMatrixImage, metaPrompt, styleAnalysis }

// 情绪板
GeneratedAsset { id, productName, axis, imageUrl, promptData, timestamp }

// CMF引擎
GeneratedAsset { id, layoutId, url, timestamp }
```

整合需要定义统一的 `DesignJob` 数据模型。

### 4.2 存储方案碎片化

| 引擎 | 存储 | 限制 |
|------|------|------|
| KV | IndexedDB | 本地，无跨设备 |
| 矩阵 | Firebase Firestore | 需要 Auth，有云端依赖 |
| 情绪板 | IndexedDB | 本地，无跨设备 |
| CMF | localStorage | 极易超配额，最不稳定 |

**最严重的问题：** 把 base64 图片存在 localStorage / IndexedDB 本身就是错误的工程决策，大量图片会导致存储爆炸。生产系统应将图片写入文件系统或对象存储，数据库只存路径。

### 4.3 API 调用成本和并发控制

一张输入图触发所有四个引擎：
```
KV主视觉:  6次 API 调用 → 4张图
细节矩阵:  2次 API 调用 → 1张图
情绪板:    9次 API 调用 → 4张图
CMF叙事:   4次 API 调用 → 4张图
─────────────────────────────────
合计:     21次 API 调用 → 13张图（每张输入图）
```

当前所有引擎都是**串行**处理，21次调用全部顺序执行。整合后需要设计合理的**并发策略**，否则单张图的等待时间可能超过 10 分钟。

### 4.4 参考图策略不统一

- KV 引擎：参考图参与**渲染阶段**（影响最终图像风格）
- 情绪板：参考图参与**分析阶段**（影响 prompt 生成）
- 矩阵和 CMF：完全不支持参考图

整合平台需要明确参考图的作用域：是全局作用于所有引擎，还是可以按引擎单独配置？

### 4.5 模型配置分散

```
gemini-3.1-pro-preview      → 文本分析（3 个引擎使用）
gemini-3.1-flash-image-preview → 图像生成主力（4 个引擎均使用）
gemini-2.5-flash-image      → 降级 fallback（2 个引擎使用）
gemini-3-flash-preview      → CMF 引擎备用（未实际使用）
```

模型 ID 分散在各 `gemini.ts` 文件中，升级维护成本高。

## 五、整合平台架构设想

### 5.1 统一数据模型

```typescript
// 一个"设计任务"（单张输入图的完整出图任务）
interface DesignJob {
  id: string
  createdAt: number
  status: JobStatus

  // 输入
  input: {
    productImage: string          // 文件路径（不存 base64）
    productDescription?: string   // 可选文字描述
    referenceImages: string[]     // 参考图路径数组
  }

  // 四个子任务的输出
  outputs: {
    kv: KVOutput | null           // 主视觉
    matrix: MatrixOutput | null   // 细节矩阵
    moodboard: MoodboardOutput | null // 情绪板
    cmf: CMFOutput | null         // CMF变体
  }

  // 中间分析结果（可复用）
  analysis: {
    visualDNA?: string            // KV引擎的诊断
    styleAnalysis?: StyleAnalysis // 细节矩阵的风格分析
    moodboardAxes?: MoodboardPrompt[] // 情绪板的四轴
  }
}
```

### 5.2 能力服务层（关键：分析结果可复用）

```
┌────────────────────────────────────────────┐
│              统一分析服务                    │
│  analyzeDesign(image, refs) → DesignDNA    │
│  （合并3个引擎的分析阶段为1次 Pro 调用）      │
└───────────────────┬────────────────────────┘
                    │ DesignDNA
        ┌───────────┼───────────┐──────────┐
        ↓           ↓           ↓          ↓
  KV生成服务   矩阵生成服务  情绪板服务  CMF生成服务
  renderKV()  renderMatrix() renderMoodboard() renderCMF()
  4张         1张            4张               4张
```

**核心优化**：当前每个引擎都有自己的分析阶段，整合后**合并为一次分析调用**，从 21 次压缩到约 16 次，同时四个生成任务**并行执行**，整体时间大幅缩短。

### 5.3 工程化要求清单

| 类别 | 要解决的问题 | 方案方向 |
|------|------------|---------|
| **存储** | base64 存数据库 | 图片写本地文件系统，DB只存路径 |
| **存储** | 四套存储碎片化 | 统一用 IndexedDB + 本地图片目录 |
| **并发** | 全串行等待太久 | 分析阶段串行，四个生成任务并行 |
| **容错** | 单点失败中断全流程 | 每个子任务独立重试，互不影响 |
| **进度** | 缺乏整体进度反馈 | 统一进度总线，实时显示 21 步进度 |
| **配置** | API Key 依赖 window.aistudio | 统一 API Key 配置入口 |
| **模型** | 模型 ID 散落各处 | 统一 models.config.ts |
| **历史** | 四套历史面板 | 统一 Job 列表，按状态和时间排序 |
| **导出** | 各自打 ZIP | 统一批量导出，按 Job 组织目录结构 |

## 六、推荐技术栈

| 层 | 选择 | 理由 |
|----|------|------|
| 框架 | Next.js（App Router）| 与 utoo 官网同栈，可集成；SSR 对大图有优势 |
| 状态 | Zustand | 比 useState 更适合跨组件的 Job 状态管理 |
| 存储 | better-sqlite3 + 本地文件 | 图片存磁盘，元数据存 SQLite，轻量无依赖 |
| 任务队列 | 自研简单队列（async/await + 并发限制） | 不需要引入 Bull/BullMQ 这样的重型方案 |
| UI | 继承现有设计风格（黑色背景 + 白色字体 + Lucide 图标） | 保持视觉一致性 |

> 注：如果要部署为 Web 服务（多用户），则存储替换为 PostgreSQL + S3/OSS。如果是本地桌面工具，SQLite + 本地文件即可。

## 七、下一步行动建议

**阶段一（整合核心能力）**
1. 定义统一 `DesignJob` 数据模型
2. 创建统一 `gemini/` 服务层，合并四套 gemini.ts
3. 实现统一分析函数 `analyzeDesign()`（整合三个引擎的分析 prompt）
4. 实现四个并行生成服务，共用 `DesignDNA` 分析结果

**阶段二（工程化界面）**
5. 统一上传入口（产品图 + 可选参考图 + 可选文字描述）
6. 统一进度看板（4个子任务独立显示状态）
7. 统一结果展示（按类型分区：KV / 矩阵 / 情绪板 / CMF）
8. 统一存储和历史面板

**阶段三（生产化）**
9. 图片落盘（停止在内存/数据库中存 base64）
10. 错误恢复（子任务失败后可单独重试，不用重跑全流程）
11. 批量导出（按 Job 组织的 ZIP，含 prompt 信息）
